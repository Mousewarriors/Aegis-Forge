# Aegis Forge: Full PyRIT Integration Specification

Status: Draft  
Last updated: 2026-03-11  
Scope: `aegis-forge` PyRIT integration design and implementation reference

## 1. Purpose

This document defines the target state for a full, useful PyRIT integration inside Aegis Forge.

The goal is not to expose a thin curated subset forever, and it is also not to dump raw PyRIT internals directly into the UI with no structure. The target state is a layered integration that:

- preserves the current quick-start usability of the app
- exposes substantially more of PyRIT's real orchestration surface
- makes planned execution understandable before launch
- makes runtime behavior observable while a scan is active
- preserves findings, transcripts, and logs after stop or completion
- supports short iterative scans and longer expert runs

This file is intended to be the implementation reference for future PyRIT work.

## 2. Problem Statement

The current PyRIT integration is functional but narrow:

- the PyRIT package is installed in the runner image
- the app uses a custom wrapper around a curated scenario catalog
- only a small set of attacks, converters, and scorers are exposed
- the UI behaves more like a scenario launcher than a PyRIT orchestration surface

This leaves a gap between:

- "PyRIT is available in the app"
- and "the app exposes PyRIT as a first-class evaluator with real operator control"

The user requirement is to move toward the second state.

## 3. Design Principles

- PyRIT must be first-class beside Promptfoo and Garak, not a hidden special case.
- The default experience must remain short, comprehensible, and safe.
- Advanced control must be real. It must not be fake configurability over a tiny fixed catalog.
- Every run must show an execution plan before launch.
- Every run must emit complete logs and retain them after stop or completion.
- Findings must roll up into the same audit and reporting surfaces used elsewhere in the app.
- Expert capability should be layered behind an advanced mode, not mixed into the default quick path.

## 4. Goals

- Support both curated and advanced PyRIT workflows.
- Expose a substantially broader set of PyRIT primitives than the current wrapper.
- Provide dynamic discovery of available attacks, converters, scorers, and targets where feasible.
- Allow operators to configure short scans for rapid fix-and-retest loops.
- Allow operators to configure deeper orchestrated runs for later validation.
- Make transformed prompts, scoring strategy, and attack chain visible before execution.
- Preserve all runtime events, logs, transcripts, and reports.
- Keep the backend contract consistent with the existing `/eval/*` model.

## 5. Non-Goals

- Expose arbitrary Python object construction directly from the browser.
- Support every PyRIT class on day one.
- Replace Promptfoo or Garak.
- Build a raw notebook-like PyRIT IDE inside the app.
- Implement cloud-only PyRIT backends as a mandatory dependency.

## 6. Current State

Current implementation characteristics:

- PyRIT runner image installs `pyrit==0.11.0`.
- The app launches a custom `run_pyrit.py` wrapper.
- The wrapper currently supports:
  - `PromptSendingAttack`
  - `FlipAttack`
  - `Base64Converter`
  - `ROT13Converter`
  - `UnicodeConfusableConverter`
  - `ZeroWidthConverter`
  - `SelfAskRefusalScorer`
  - `TrueFalseInverterScorer`
  - `OpenAIChatTarget`
  - in-memory memory backend
- The frontend currently uses a curated scenario catalog.
- The current preview shows raw objectives, not transformed or attack-specific preview output.

This is a valid starting point, but it is not a full technical integration of PyRIT as an orchestration framework.

## 7. Target State Summary

The target integration has three operator modes.

### 7.1 Quick Mode

Purpose:

- short scans
- curated presets
- safe defaults
- fix-and-retest workflow

Characteristics:

- recommended scenarios only
- simple caps on objectives, attempts, turns, and runtime
- minimal required choices
- clear execution preview

### 7.2 Advanced Mode

Purpose:

- real operator control
- scenario composition
- selection of attacks, converters, scorers, and targets from supported registries

Characteristics:

- operator chooses attack type, converter chain, scorer, target profile, memory mode, retry policy, and concurrency
- app validates combinations before launch
- app shows transformed prompt samples and estimated cost

### 7.3 Expert Mode

Purpose:

- expose near-native PyRIT orchestration for expert users

Characteristics:

- load a PyRIT run configuration generated by the app
- optionally edit advanced JSON/YAML fields in a guarded editor
- run native scenario definitions that the backend recognizes as supported
- full transcript and event inspection

Expert mode is not required for the first implementation phase, but the architecture must leave room for it.

## 8. Functional Requirements

### 8.1 Discovery and Catalog

The app must distinguish between:

- curated presets
- supported PyRIT primitives
- installed but unsupported primitives

The backend must maintain or generate a PyRIT registry that includes:

- attacks
- prompt converters
- scorers
- prompt targets
- memory backends
- built-in or app-defined scenario templates

Each registry item must expose:

- stable id
- human label
- type
- description
- supported parameters
- compatibility constraints
- readiness status
- whether it is recommended, experimental, or hidden

The backend may initially use a curated manifest with introspection-assisted validation, then evolve toward fuller discovery.

### 8.2 Execution Planning

Before launch, the operator must be shown a deterministic execution plan that answers:

- what attack or scenario will run
- against which target
- with which converter chain
- with which scorer
- how many objectives or prompts will be used
- how many attempts are expected
- how many turns are allowed
- whether retries are enabled
- how memory is configured
- whether the run is local-only or depends on external services
- what transformed prompt samples look like

The plan must be available via backend preflight and shown in the UI before a run starts.

### 8.3 Run Controls

The PyRIT UI must support at minimum:

- objective count per scenario
- prompt or attempt cap per objective
- max turns
- concurrency
- retries
- timeout per request
- overall runtime cap
- target model selection
- scorer model selection
- target endpoint profile
- converter chain order
- memory mode
- protection on or off
- run label
- stop behavior

The UI must also support run presets:

- quick refusal check
- encoding sweep
- mutation sweep
- multi-turn jailbreak check
- long-form deep run

### 8.4 Observability

The app must provide complete PyRIT observability during and after a run.

Required views:

- execution plan
- live event stream
- per-scenario progress
- per-attempt status
- transformed prompt preview
- full response transcript
- scorer verdict and reason
- run summary
- retained logs after stop or completion

The live event stream must not be truncated to a small fixed count in the backend. The UI may paginate or virtualize, but the data must remain available.

### 8.5 Findings and Reporting

PyRIT results must feed the same reporting concepts used elsewhere in the app:

- vulnerability matrix
- flagged findings list
- raw transcripts
- audit feed
- run report
- exportable artifacts

Every failed or vulnerable attempt must be visible in one consolidated place with:

- scenario id
- attack type
- transformed prompt or original prompt
- response
- scorer rationale
- severity or confidence if available

### 8.6 Retention

The app must retain:

- provisional reports for active and stopped runs
- official reports for completed runs
- full event logs
- recent attempts
- raw JSONL event output
- final normalized report payload

Stop must not erase the evidence surface.

### 8.7 Safety and Validation

The app must validate operator input before launch.

Examples:

- unsupported attack and scorer combinations
- unsupported converter chain order
- expert-mode config fields that the backend cannot safely construct
- unsupported target backends in the current environment

The backend must reject invalid combinations with explicit reasons.

## 9. Architecture

### 9.1 Frontend

The frontend should evolve from a flat scenario selector into a layered PyRIT workbench.

Recommended sections:

- evaluator header
- mode switch: Quick / Advanced / Expert
- scenario or attack builder
- execution plan preview
- transformed prompt preview
- run controls
- live activity
- findings
- report

### 9.2 Backend

The backend remains the source of truth for:

- registry generation
- preflight and validation
- config assembly
- run lifecycle
- event normalization
- report generation
- retention

The backend should not let the frontend invent PyRIT object graphs on its own.

### 9.3 Runner

The PyRIT runner should remain a separate execution layer, but it must become more generic.

The runner should support:

- parameterized attack construction
- parameterized converter chains
- scorer selection
- target profile selection
- memory backend selection
- event emission for every major lifecycle step

The runner must emit structured JSONL events, not freeform logs only.

## 10. Proposed Backend Components

Recommended new backend modules:

- `backend/pyrit_registry.py`
  - supported attack, converter, scorer, target, and memory registry
- `backend/pyrit_introspection.py`
  - optional runtime discovery and validation against installed PyRIT
- `backend/pyrit_config_builder.py`
  - constructs normalized runner config from UI requests
- `backend/pyrit_report_parser.py`
  - parses raw runner output into normalized app results
- `backend/pyrit_presets.py`
  - curated quick-mode templates

The current logic inside `backend/main.py` should gradually move into dedicated modules as PyRIT grows.

## 11. Proposed API Surface

The app can continue using `/eval/*`, but PyRIT-specific sub-endpoints should be added for clarity.

Recommended additions:

- `GET /eval/pyrit/catalog`
  - returns supported attacks, converters, scorers, targets, memory backends, presets
- `POST /eval/pyrit/preflight`
  - validates config and returns execution plan
- `POST /eval/pyrit/transform-preview`
  - returns raw objective plus transformed samples for the selected converter chain or attack
- `POST /eval/pyrit/config/validate`
  - validates advanced or expert JSON payloads
- `GET /eval/pyrit/run/{run_id}/plan`
  - returns the saved execution plan
- `GET /eval/pyrit/run/{run_id}/events`
  - raw normalized event feed
- `GET /eval/pyrit/run/{run_id}/transcripts`
  - normalized transcripts by scenario and attempt

The generic endpoints should remain:

- `/eval/run`
- `/eval/status/{run_id}`
- `/eval/logs/{run_id}`
- `/eval/report/{run_id}`
- `/eval/stop/{run_id}`

## 12. PyRIT Request Model

The current `EvalRequest` is not rich enough for full PyRIT control. A dedicated PyRIT request model should be introduced.

Recommended fields:

- `mode`
- `preset_id`
- `selected_scenarios`
- `attack_id`
- `attack_params`
- `converter_chain`
- `converter_params`
- `scorer_id`
- `scorer_params`
- `target_profile_id`
- `target_params`
- `memory_backend`
- `memory_params`
- `objective_templates`
- `objective_limit`
- `attempt_cap`
- `max_turns`
- `concurrency`
- `retry_limit`
- `request_timeout_seconds`
- `runtime_cap_seconds`
- `session_hardened`
- `proxy_enabled`
- `label`
- `notes`

This model can be mapped into a normalized runner config.

## 13. Runner Configuration Model

The runner config should be explicit and serializable. It should not rely on implicit defaults hidden inside the frontend.

Suggested top-level structure:

- run metadata
- environment metadata
- target definition
- scorer definition
- memory definition
- scenario list
- execution policy
- retention settings

Each scenario entry should contain:

- scenario id
- display label
- attack definition
- converter chain
- objective list
- max turns
- attempt cap
- stop conditions

## 14. Event Model

The runner must emit structured events for:

- `run_start`
- `plan_snapshot`
- `scenario_start`
- `objective_start`
- `prompt_transformed`
- `attempt`
- `score_result`
- `scenario_complete`
- `warning`
- `error`
- `summary`
- `fatal`

Each event should include:

- run id
- timestamp
- evaluator
- scenario id
- attempt index if applicable
- event type
- structured payload

For `attempt` events, include:

- raw objective
- transformed prompt if available
- original prompt if different
- target response
- score
- pass or fail
- reason
- turn count
- execution duration

## 15. UI Specification

### 15.1 Quick Mode

Quick mode should show:

- preset cards
- scenario count
- objective cap
- estimated attempts
- transformed preview sample
- launch button

The operator should be able to run a useful refusal check in under one minute.

### 15.2 Advanced Mode

Advanced mode should show:

- attack selector
- converter chain builder
- scorer selector
- target selector
- memory selector
- objective editor
- runtime controls
- execution plan panel

The key constraint is that every selection updates the execution plan immediately.

### 15.3 Expert Mode

Expert mode should show:

- generated normalized config
- read-only by default
- optional guarded JSON editor
- validation feedback
- execution plan diff if the config changes

Expert mode should be clearly labeled as advanced and unsupported combinations should fail at validation time.

## 16. Preview Requirements

The current PyRIT preview is insufficient because it shows only raw objectives.

The improved preview must show:

- raw objective
- attack type
- converter chain
- transformed prompt sample if deterministic
- note when transformation occurs at runtime and cannot be fully previewed

Examples:

- Base64 and ROT13 can show exact transformed samples
- Unicode confusable and zero-width can show rendered and escaped views
- FlipAttack can show attack-specific mutation preview if deterministically reproducible

## 17. Findings Model

PyRIT findings should use the same normalized result model as other evaluators where possible.

Every normalized PyRIT result should contain:

- `plugin_id` or `scenario_id`
- `plugin_label`
- `category`
- `attack_type`
- `converter_chain`
- `prompt`
- `original_prompt`
- `response`
- `passed`
- `reason`
- `score`
- `status_text`
- `timestamp`
- `attempt_index`
- `turns_executed`
- `execution_time_seconds`

Additional PyRIT-specific fields may be stored in `metadata`.

## 18. Audit Feed Integration

PyRIT must contribute to the same audit feed as Promptfoo and Garak.

Minimum audit record types:

- run launched
- scenario started
- vulnerable attempt observed
- scenario completed
- run stopped
- run completed

The home dashboard should surface PyRIT findings without requiring the user to open the dedicated evaluator page.

## 19. Reporting Requirements

The final report should include:

- run summary
- scenario coverage
- findings table
- raw transcripts
- blocked versus vulnerable ratios
- scorer rationale
- run configuration snapshot

The report must not disappear after stop.

The report should support:

- provisional state during execution
- official state after completion
- stopped state with retained evidence

## 20. Memory Backend Strategy

PyRIT memory support should be phased.

Phase 1:

- in-memory only

Phase 2:

- local SQLite backend

Phase 3:

- optional cloud or remote memory backends if needed

The UI must show which memory mode is active and whether history persists across runs.

## 21. Compatibility Strategy

Not every PyRIT primitive should be exposed immediately.

Each registry item must have a support state:

- supported
- experimental
- unavailable in this environment

Support state should depend on:

- installed package version
- app adapter support
- target adapter support
- scorer model availability
- environment readiness

## 22. Security Constraints

The backend must never allow the browser to specify arbitrary Python import paths for execution.

Instead:

- the backend exposes a validated registry of allowed classes or scenario templates
- expert config editing still maps through registry validation
- unsupported classes are rejected

This keeps the app operable without turning it into a remote Python execution surface.

## 23. Performance Requirements

The app must support both short and deep runs.

Short run goals:

- launch to first result in under 30 seconds in local-ready conditions
- one-scenario refusal check in under 2 minutes

Deep run goals:

- allow longer multi-turn or multi-scenario runs
- maintain full event retention
- avoid UI slowdown by paginating or virtualizing logs

## 24. Rollout Plan

### Phase 1: Strengthen Current Curated Integration

- improve preview to show transformed samples
- add richer event model
- retain full logs and transcripts
- improve findings normalization
- add run plan persistence

### Phase 2: Introduce Advanced Builder

- create backend registry
- expose attacks, converters, scorers, targets, memory modes
- implement advanced UI composer
- validate combinations in preflight

### Phase 3: Expand Supported PyRIT Surface

- add more PyRIT attacks
- add more scorers
- add multi-turn support where viable
- add SQLite memory backend

### Phase 4: Expert and Native-Oriented Mode

- config editor
- import or generate richer run definitions
- broader scenario support
- optional export and replay features

## 25. Acceptance Criteria

The integration is considered complete enough for practical use when all of the following are true.

- An operator can run a quick PyRIT check in fewer than five clicks.
- An operator can build a custom attack with chosen attack, converters, scorer, and target.
- The preflight plan shows the exact planned configuration and estimated runtime.
- The preview shows transformed samples when possible.
- A running scan exposes real progress and full logs.
- Stopped scans retain logs, findings, and provisional reports.
- Completed scans retain official reports and raw transcripts.
- Failed or vulnerable attempts appear in the findings surface and vulnerability matrix.
- The backend validates and rejects unsupported combinations with explicit reasons.
- The implementation is covered by backend and frontend regression tests.

## 26. Open Questions

- Which additional PyRIT attacks should be first-class in Phase 2?
- Should Expert mode allow JSON editing only, or YAML too?
- Do we want replay support from saved runner configs?
- Should transformed prompt previews be generated in backend-only logic for consistency?
- When multi-turn support is added, should the UI show transcript trees or flat turn lists?

## 27. Recommended Implementation Order

Recommended next implementation order:

1. Improve preview semantics and event richness for the current curated integration.
2. Introduce a backend PyRIT registry and move PyRIT logic out of `main.py`.
3. Add the Advanced PyRIT builder UI.
4. Expand supported PyRIT primitives and memory modes.
5. Add Expert mode once the registry and validation layers are stable.
